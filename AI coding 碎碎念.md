# AI coding 碎碎念


## 为什么要AI coding? 
AI 现在如火如荼，自己在工作中确实也遇到了一些困难。之前每周五我会在系统上筛选不同批次，不同资源类型，进行工包管理
人工肉眼判断该分配哪些工包，这也就产生了readme中所说的问题。

隔壁项目组小伙伴使用在线excel搭建了一个按照批次展示各个工包，每个工包计算完成率。excel搭建的好处是
能把工包汇总在一起，方便工包分配人查看。

但这也带来了一些问题，一是需要人工复制粘贴进去对应批次的工包，
二是也还需要人工肉眼判断优先分配哪些。

我看到他的这个在线excel后，提出如果能够直接爬取网站上的所有工包，直接写入excel展示就好了，这样不用人工复制粘贴，减少出错机会。可以避免问题一。但是因为是公司内网系统，而且需要账号验证，爬虫应该是有些困难。后来我想到之前每次做工包汇总分析时都会下载excel文件。那我只需要对excel文件里面的数据进行提取，汇总分析，形成我想要的展示形式即可。

那么用什么形式呢？其实excel本身也很强大，利用一些公式完全可以做到。但是我们公司内网使用的是wps版本的excel，有些excel的高级公式和功能是没有办法使用的。

我能想到的最好就是简单，轻量，稳定。于是我就开始了AI coding 之路

## AI coding总结

1. 最初需求/功能不清晰没有关系，可以和ai交流，交互一步步朝着自己想要的方向走，有些可能最开始想的展现形式，随着开发过程也会有变化，变成更好用的方式
2. ai 的理解能真得越来越强，有些我现在都不太能理解的自己说出来的需求，ai能先按照它的理解做一个样子，往往就是这种方式，能引导出我们想要什么。我们可以直接看到ai做的东西，不断纠正
3. 给ai的指令越清晰，它执行的就越好，当然不清晰也没关系，参考第1条。最终总能清晰的。
4. ai coding 的全流程真得还挺像把大象装进冰箱需要几步？第一步打开冰箱门，第二步把大象放进去，第三步把冰箱门关上。任何复杂事物其实都能拆解，不断拆解。拆解这三大步后，每一步还能再明确，比如第一步的冰箱门如何打开？是单开门打开还是双开门打开？是十字门还是法式门？

5. ai coding 时代，感觉对产品经理还是很友好的，程序员就要有危机感了，也感受到coding可能未来不会作为一个职业了（当然特别高级的研究模型算法的还是很牛）普通的开发可能会是人人都会的一项技能了。那重要的其实是你能用这项技能解决什么问题？
